# 测试用例设计文档

## 一、测试范围

本项目针对 GitHub API 和 GitHub Web 进行自动化测试，覆盖以下功能模块：

### API 测试范围

| 模块 | 测试点 | 优先级 |
|------|--------|--------|
| 用户功能 | 获取用户信息、获取认证用户、获取仓库列表、无效用户测试 | P0 |
| 仓库功能 | 获取仓库详情、处理不存在仓库 | P0 |
| 搜索功能 | 搜索仓库、空结果处理 | P1 |
| Issue 功能 | 获取 Issue 列表、创建 Issue、状态筛选 | P1 |
| 频率限制 | 获取 API 频率限制状态 | P2 |
| 错误响应 | HTTP 错误响应测试（404、401、403、500等） | P1 |
| 高级特性 | 熔断器、限流器、插件系统、安全模块 | P1 |
| 性能测试 | 响应时间、并发请求 | P2 |
| 压力测试 | 高并发、持续压力、负载测试 | P2 |
| 配置验证 | 环境配置、日志级别、浏览器配置 | P2 |
| 数据生成 | 用户/仓库/Issue 数据生成配置 | P2 |

### Web UI 测试范围

| 模块 | 测试点 | 优先级 |
|------|--------|--------|
| 登录功能 | 登录成功、登录失败、表单验证 | P0 |
| 首页功能 | 页面加载、导航、搜索、仓库列表 | P1 |
| 用户操作 | 个人中心、仓库创建、设置导航 | P1 |
| 仓库页面 | 创建仓库数据、页面元素定位 | P1 |
| 通用元素 | 加载动画、提示消息、下拉菜单 | P2 |
| 配置测试 | 超时配置、截图配置 | P2 |

## 四、测试策略

### 4.1 Mock 测试策略

由于 GitHub API 有频率限制，本项目采用 Mock 测试策略：

1. **单元测试**: 使用 Mock 模拟 API 响应，测试框架逻辑
2. **集成测试**: 使用 Mock 模拟完整流程，测试模块间交互
3. **边界测试**: 使用 Mock 模拟各种边界情况和异常场景

### 4.2 测试优先级

- **P0**: 核心功能，必须通过
- **P1**: 重要功能，应该通过
- **P2**: 次要功能，建议通过

### 4.3 测试覆盖率目标

| 指标 | 目标 |
|------|------|
| API 层覆盖率 | >= 95% |
| 公共组件覆盖率 | >= 80% |
| 用例通过率 | 100% |
| 数据使用率 | 100% |

## 五、测试执行

### 5.1 执行命令

```bash
# 运行所有测试
pytest

# 运行 API 测试
pytest test_cases/api/ -v

# 运行 Web UI 测试
pytest test_cases/web/ -v

# 生成 Allure 报告
pytest --alluredir=./allure-results
allure serve allure-results
```

### 5.2 CI/CD 集成

项目使用 GitHub Actions 进行持续集成，每次提交自动运行测试。
